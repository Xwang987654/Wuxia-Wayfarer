# 技术架构设计

## 状态：待定

技术栈尚未最终确定。本文档记录候选方案和评估标准，待决策后更新。

## 候选技术栈

### 方案 A：Web 技术（HTML/CSS/JS + Canvas/WebGL）

**优势**
- 跨平台（浏览器直接运行）
- 开发迭代快
- 部署简单（静态文件托管）
- 社区资源丰富

**劣势**
- 性能受限（大量精灵/粒子时）
- 移动端适配需额外工作
- 无原生桌面功能

**适合框架**
- Phaser.js — 2D游戏专用框架，社区活跃
- PixiJS — 渲染引擎，灵活度高
- 纯 Canvas API — 最轻量，但工作量大

### 方案 B：Godot

**优势**
- 开源免费，无授权费
- 2D支持优秀
- 内置场景/节点系统适合游戏开发
- 可导出多平台（PC/Mac/Web/移动端）

**劣势**
- 需要学习 GDScript 或 C#
- 编辑器较重
- 社区相比Unity较小

### 方案 C：Unity

**优势**
- 生态最成熟，资源商店丰富
- C# 开发，类型安全
- 多平台导出

**劣势**
- 对2D游戏来说偏重
- 授权费用（虽有免费版）
- 编辑器资源占用大

## 技术选型标准

| 标准 | 权重 |
|------|------|
| 2D/像素渲染性能 | 高 |
| 开发效率（一人开发） | 高 |
| 学习成本 | 高 |
| 部署便利性 | 中 |
| 多平台支持 | 中 |
| 社区/文档质量 | 中 |
| 扩展性（后续功能） | 低（MVP阶段） |

## 架构分层（通用）

无论选择哪种技术栈，游戏架构分为以下几层：

```
┌─────────────────────────────────────┐
│           表现层（Rendering）         │
│  场景渲染 · UI渲染 · 动画 · 粒子     │
├─────────────────────────────────────┤
│           交互层（Input）             │
│  点击/触摸 · 菜单选择 · 快捷键       │
├─────────────────────────────────────┤
│           游戏逻辑层（Game Logic）    │
│  时间系统 · 战斗计算 · 经济系统       │
│  NPC行为 · 任务系统 · 声望系统       │
├─────────────────────────────────────┤
│           数据层（Data）              │
│  存档/读档 · 配置表 · 存量数据        │
└─────────────────────────────────────┘
```
